# Fix n11 while-if-while-break decompilation failure Spec

## Why
When decompiling `while x < limit: if x % 2 == 0: y = 0; while y < 10: if y == 5: break; y += 1; a = x; x += 1`, the decompiler incorrectly produces code where the inner while loop is not nested inside the `if x % 2 == 0` block. The root cause has two parts:
1. Block 14 (`if x % 2 == 0`) is the outer loop's `header_block` and is incorrectly skipped as an IfRegion candidate by the `has_body_stmt` check in `_check_elif_chain`, because the `BINARY_OP` (modulo `x % 2`) is part of the condition expression, not a standalone body statement.
2. In `_process_if_blocks`, the CONTINUE role check runs before the nested region entry check, causing block 32 (the inner while loop entry) to be processed as a continue statement instead of being recognized as a nested LoopRegion entry.
3. The natural back edge block (62) gets marked as generated by IfRegion:48 before the loop body generator can process it, causing the `y += 1` statement to be lost.

## What Changes
- **Fix 1**: Remove `BINARY_OP` from the `has_body_stmt` check in `_check_elif_chain` (region_analyzer.py ~line 8726). `BINARY_OP` is commonly part of condition expressions (e.g., `x % 2` in `if x % 2 == 0`) and should not prevent IfRegion creation. Augmented assignments like `x += 1` are still caught because they also have `STORE_FAST` instructions.
- **Fix 2**: In `_process_if_blocks` (region_ast_generator.py ~line 5277), add `_is_structural_entry` check before the CONTINUE role handling. When a block is the entry of a nested LoopRegion/TryExceptRegion/WithRegion/MatchRegion, skip the CONTINUE role check and let the nested region check handle it.
- **Fix 3**: In `_loop_generate_body` (region_ast_generator.py ~line 2300), process the natural back edge BEFORE iterating over other body blocks. This prevents the back edge from being marked as generated by other regions (like IfRegion) before the loop body generator can process it.

## Impact
- Affected code: `core/cfg/region_analyzer.py` — `_check_elif_chain` method; `core/cfg/region_ast_generator.py` — `_process_if_blocks` and `_loop_generate_body` methods
- Affected tests: `tests/exhaustive/nested/test_n11while_if_while_break_a_b.py` and `test_n11while_if_while_break_n_m.py` (now passing)
- Regression result: 76f/1734p (improved from baseline 77f/1730p — 1 fewer failure, 4 more passes)

## ADDED Requirements

### Requirement: Loop header block with condition expression must be eligible as IfRegion
When a block is a loop's `header_block` AND the loop has a separate `condition_block`, the block's forward conditional jump is an `if` guard inside the loop. The `has_body_stmt` check SHALL NOT prevent creating an IfRegion for such a block when the "body statements" are `BINARY_OP` instructions that are part of the condition expression (not standalone body code).

#### Scenario: n11 while-if-while-break with constants
- **WHEN** decompiling `while x < limit: if x % 2 == 0: y = 0; while y < 10: if y == 5: break; y += 1; a = x; x += 1` with a=5, b=10
- **THEN** the decompiled bytecode instruction count matches the original

#### Scenario: n11 while-if-while-break with variables
- **WHEN** decompiling the same pattern with variables n, m
- **THEN** the decompiled bytecode instruction count matches the original

### Requirement: Structural region entries must bypass CONTINUE role handling
When a block is the entry of a nested LoopRegion/TryExceptRegion/WithRegion/MatchRegion, the CONTINUE role check SHALL NOT apply. The block SHALL be handled by the nested region check instead.

### Requirement: Natural back edge must be processed before other body blocks
In `_loop_generate_body`, the natural back edge SHALL be processed before iterating over other body blocks. This ensures the back edge is not prematurely marked as generated by other regions.

## MODIFIED Requirements

### Requirement: has_body_stmt check in _check_elif_chain shall not count BINARY_OP as body statement
The `has_body_stmt` check SHALL exclude `BINARY_OP` from the body statement list. `BINARY_OP` is commonly part of condition expressions and should not prevent IfRegion creation. Augmented assignments like `x += 1` are still caught because they also have `STORE_FAST` instructions.
