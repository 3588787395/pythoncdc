# ADR-003: 为什么每个区域类型只有一个协调器

## 状态
✅ 已采纳

## 背景
在 `region_ast_generator.py` 中，需要将区域分析器识别出的14种区域类型转换为对应的AST节点。早期实现中存在以下问题：

1. **方法职责不清**: 一个 `_generate_region()` 方法处理所有区域类型，内部包含大量的if-elif分支
2. **代码重复**: 不同区域类型的处理逻辑分散在多个方法中，存在重复代码
3. **难以扩展**: 添加新区域类型时需要在多处修改代码
4. **测试困难**: 无法独立测试某种区域类型的生成逻辑

## 决策
**遵循"一区域一方法"原则：每种区域类型对应唯一的AST生成协调器方法。**

### 14种区域类型的完整映射表

| 区域类型 (RegionType) | 协调器方法 | 生成的AST结构 |
|----------------------|-----------|--------------|
| `SEQUENCE` | `_handle_sequence_region()` | 语句列表 |
| `IF_THEN` | `_handle_if_then_region()` | If节点（无else） |
| `IF_ELSE` | `_handle_if_else_region()` | If节点（有else） |
| `IF_ELIF_CHAIN` | `_handle_if_elif_chain_region()` | 多层If/Elif链 |
| `FOR_LOOP` | `_handle_for_loop_region()` | For循环 |
| `WHILE_LOOP` | `_handle_while_loop_region()` | While循环 |
| `TRY_EXCEPT` | `_handle_try_except_region()` | Try/Except |
| `TRY_FINALLY` | `_handle_try_finally_region()` | Try/Finally |
| `WITH` | `_handle_with_region()` | With语句 |
| `MATCH` | `_handle_match_region()` | Match语句 |
| `ASSERT` | `_handle_assert_region()` | Assert语句 |
| `BOOL_OP` | `_handle_bool_op_region()` | 布尔运算表达式 |
| `TERNARY` | `_handle_ternary_region()` | 三元表达式 |
| `COMPREHENSION` | `_handle_comprehension_region()` | 推导式 |

### 标准协调器模式

```python
class RegionASTGenerator:
    def _dispatch_region(self, region: Region) -> List[Dict]:
        """统一分发入口"""
        handler = self._get_handler(region.region_type)
        return handler(region)

    def _get_handler(self, region_type: RegionType):
        """获取对应的处理方法"""
        handlers = {
            RegionType.SEQUENCE: self._handle_sequence_region,
            RegionType.IF_THEN: self._handle_if_then_region,
            RegionType.IF_ELSE: self._handle_if_else_region,
            # ... 其他映射
        }
        return handlers[region_type]

    def _handle_if_else_region(self, region: IfRegion) -> List[Dict]:
        """
        处理if-else区域的专用协调器
        职责：
        1. 提取条件表达式
        2. 递归生成then-body和else-body
        3. 组装为If AST节点
        """
        condition = self._extract_condition(region.condition_block)
        then_body = self._generate_body(region.then_blocks)
        else_body = self._generate_body(region.else_blocks)

        return [{
            'type': 'If',
            'test': condition,
            'body': then_body,
            'orelse': else_body
        }]
```

## 理由

### 1. 单一职责原则 (SRP)
每个协调器方法只负责一种区域类型的转换：

```python
# ✅ 清晰的职责划分
def _handle_for_loop_region(self, region: LoopRegion):
    """只关注for循环的转换逻辑"""
    target = self._extract_iter_target(region.init_block)
    iter_expr = self._extract_iterator(region.condition_block)
    body = self._generate_body(region.body_blocks)
    orelse = self._generate_body(region.else_blocks)

    return [{'type': 'For', 'target': target, 'iter': iter_expr, 'body': body, 'orelse': orelse}]

def _handle_while_loop_region(self, region: LoopRegion):
    """只关注while循环的转换逻辑"""
    condition = self._extract_condition(region.condition_block)
    body = self._generate_body(region.body_blocks)
    orelse = self._generate_body(region.else_blocks)

    return [{'type': 'While', 'test': condition, 'body': body, 'orelse': orelse}]
```

修改for循环逻辑不会影响while循环，反之亦然。

### 2. 可测试性增强
可以针对每种区域类型编写独立的单元测试：

```python
class TestIfElseHandler(unittest.TestCase):
    def test_simple_if_else(self):
        region = create_if_else_region(...)
        generator = RegionASTGenerator(cfg)
        result = generator._handle_if_else_region(region)
        self.assertEqual(result[0]['type'], 'If')
        self.assertIn('orelse', result[0])

class TestForLoopHandler(unittest.TestCase):
    def test_for_with_else(self):
        region = create_for_loop_region(...)
        generator = RegionASTGenerator(cfg)
        result = generator._handle_for_loop_region(region)
        self.assertEqual(result[0]['type'], 'For')
```

测试覆盖率从40%提升至70%+的重要推动力。

### 3. 扩展性保证
添加新区域类型的标准流程：

1. 在 `RegionType` 枚举中添加新类型
2. 实现对应的检测逻辑（在 `region_analyzer.py` 中）
3. 添加新的协调器方法（遵循命名规范 `_handle_xxx_region`）
4. 在映射表中注册

不需要修改任何已有的协调器方法。

### 4. 共享逻辑的正确处理方式
当多种区域类型需要共享某些逻辑时，使用**辅助方法**而非重复协调器：

```python
# ❌ 错误：创建多个类似的协调器
def _handle_try_except_v1(self, region): ...
def _handle_try_except_v2(self, region): ...  # 针对Python 3.11

# ✅ 正确：通过辅助方法复用逻辑
def _handle_try_except_region(self, region):
    base = self._build_try_base(region)
    handlers = self._build_exception_handlers(region)
    return {**base, 'handlers': handlers}

def _build_try_base(self, region):
    """共享的try基础结构构建"""
    body = self._generate_body(region.body_blocks)
    return {'type': 'Try', 'body': body}

def _build_exception_handlers(self, region):
    """共享的异常处理器构建"""
    handlers = []
    for handler in region.handlers:
        handlers.append({
            'type': 'ExceptHandler',
            'type_name': handler.exception_type,
            'body': self._generate_body(handler.body_blocks)
        })
    return handlers
```

## 违规示例与修正方案

### 违规示例1: 超级协调器

```python
# ❌ 禁止：一个方法处理所有情况
def _generate_region(self, region):
    if region.type == 'if':
        if region.has_else:
            # 50行 if-else 处理
        else:
            # 30行 if-only 处理
    elif region.type == 'loop':
        if region.is_for:
            # 60行 for 循环处理
        else:
            # 50行 while 循环处理
    elif region.type == 'try':
        # 80行 try-except 处理
    # ... 总计500+行
```

**修正**: 拆分为独立的协调器方法。

### 违规示例2: 重复协调器

```python
# ❌ 禁止：同一类型有多个处理入口
def _handle_if_region(self, region): ...  # 入口1
def _process_if_statement(self, region): ...  # 入口2（功能重叠）
def _generate_if_ast(self, region): ...  # 入口3（功能重叠）
```

**修正**: 统一为一个 `_handle_if_xxx_region()` 方法，其他方法标记为deprecated并逐步移除。

## 后果

### 正面影响
- ✅ **圈复杂度降低**: 每个方法的圈复杂度控制在5-10之间
- ✅ **代码导航清晰**: 通过方法名即可定位到特定区域类型的处理逻辑
- ✅ **Code Review效率提升**: 审查者只需关注被修改的协调器方法
- ✅ **并行开发支持**: 不同开发者可以同时修改不同的区域类型处理逻辑

### 负面影响
- ❌ **方法数量增加**: 从3-5个大方法变为14+个小方法
- ❌ **需要严格命名规范**: 必须遵循 `_handle_{type}_region()` 的命名约定
- ❌ **初期重构成本**: 将现有的大方法拆分需要仔细验证等价性
