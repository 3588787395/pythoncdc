# ADR-006: 如何处理Python版本差异（3.8/3.10/3.11）

## 状态
🔄 待定（Phase 5规划）

## 背景
Python反编译器需要支持多个Python版本的字节码。每个版本引入的新特性和字节码变更给反编译器带来了兼容性挑战：

### 各版本关键新特性

| 版本 | 关键新特性 | 字节码影响 |
|------|-----------|-----------|
| **3.8** | 海象运算符 `:=`、仅位置参数 `/` | 新增 `LOAD_METHOD`, `CALL_METHOD` |
| **3.9** | 字典合并 `|`、类型泛型内置 | 统一字典/列表操作码 |
| **3.10** | 结构化模式匹配 `match/case`、更好的错误消息 | 前向/后向跳转分离，新增 `MATCH_*` 系列 |
| **3.11** | 异常处理重构（Zero-Cost Exceptions）、Try-Star | 异常表替代栈式异常处理，新增 `PUSH_EXC_INFO`, `WITH_EXCEPT_START` |
| **3.12** | 类型参数、f-string改进、Per-Interpreter GIL | 进一步优化指令集 |

### 当前面临的挑战

1. **操作码名称变化**: 同一语义在不同版本使用不同操作码
2. **异常处理机制**: 3.11的异常表与之前版本完全不同
3. **控制流表示**: 跳转指令的方向和偏移计算方式变化
4. **新语法结构**: match-case (3.10)、try-star (3.11) 需要全新的处理逻辑

## 当前策略：条件编译 + 特性检测 + Skip标记

### 策略1: 条件编译

```python
# bytecode/python311_support.py

import sys

PY_VERSION = sys.version_info[:2]

if PY_VERSION >= (3, 11):
    EXCEPTION_TABLE_SUPPORTED = True
    NEW_JUMP_OPS = {
        'JUMP_FORWARD',
        'JUMP_BACKWARD',
        'POP_JUMP_FORWARD_IF_FALSE',
        'POP_JUMP_BACKWARD_IF_TRUE',
        # ... 更多3.11+ 操作码
    }
else:
    EXCEPTION_TABLE_SUPPORTED = False
    NEW_JUMP_OPS = set()
```

### 策略2: 运行时特性检测

```python
# core/cfg/region_analyzer.py

class RegionAnalyzer:
    def __init__(self, cfg):
        self.cfg = cfg
        self._detect_python_version_features()

    def _detect_python_version_features(self):
        """检测当前字节码使用的Python版本特性"""
        self.has_exception_table = hasattr(self.cfg.code_obj, 'co_exceptiontable')
        self.use_backward_jumps = any(
            'BACKWARD' in instr.opname
            for block in self.cfg.blocks
            for instr in block.instructions
        )

    def _analyze_exceptions(self):
        if self.has_exception_table:
            return self._analyze_exception_table_311()
        else:
            return self._analyze_exception_stack_based()
```

### 策略3: Skip标记管理

对于尚未完全支持的功能，使用skip标记：

```python
# tests/test_functional_verification.py

import pytest

PYTHON_VERSION = sys.version_info[:2]

@pytest.mark.skipif(
    PYTHON_VERSION >= (3, 11),
    reason="Exception table handling not fully supported for 3.11+"
)
def test_try_except_finally_complex():
    """复杂的try-except-finally测试"""
    source = """
    def complex_try():
        try:
            try:
                risky()
            except ValueError:
                handle_ve()
            finally:
                cleanup1()
        except TypeError:
            handle_te()
        finally:
            cleanup2()
    """
    verifier = DecompilationVerifier()
    result = verifier.verify(source)
    assert result.is_passing

@pytest.mark.skipif(
    PYTHON_VERSION < (3, 10),
    reason="Match-case only available in Python 3.10+"
)
def test_match_statement_basic():
    """基本match语句测试"""
    source = """
    def match_example(value):
        match value:
            case 0:
                return "zero"
            case 1:
                return "one"
            case _:
                return "other"
    """
    verifier = DecompilationVerifier()
    result = verifier.verify(source)
    assert result.is_passing
```

图例：
- ✅ 完全支持 (测试通过率 >85%)
- 🔄 部分支持 (测试通过率 50-85%)

- ❌ 不支持或已知问题多
- N/A 该版本不存在此功能

## 未来可能的改进方向

### 方案A: 抽象层方案（推荐用于Phase 5）

创建版本无关的字节码抽象层：

```python
# bytecode/abstract_ops.py

class AbstractOpcode(Enum):
    """版本无关的操作码抽象"""
    LOAD_CONST = "load_constant"
    BINARY_ADD = "binary_add"
    CONDITIONAL_JUMP_FORWARD = "cond_jump_forward"
    CONDITIONAL_JUMP_BACKWARD = "cond_jump_backward"
    # ... 所有抽象操作码


class VersionAwareDisassembler:
    """版本感知的反汇编器"""

    VERSION_MAPPINGS = {
        (3, 8): {
            'LOAD_CONST': AbstractOpcode.LOAD_CONST,
            'POP_JUMP_IF_FALSE': AbstractOpcode.CONDITIONAL_JUMP_FORWARD,
            # ...
        },
        (3, 11): {
            'LOAD_CONST': AbstractOpcode.LOAD_CONST,
            'POP_JUMP_FORWARD_IF_FALSE': AbstractOpcode.CONDITIONAL_JUMP_FORWARD,
            'POP_JUMP_BACKWARD_IF_FALSE': AbstractOpcode.CONDITIONAL_JUMP_BACKWARD,
            # ...
        },
    }

    def disassemble(self, code_obj, version):
        mapping = self.VERSION_MAPPINGS[version]
        abstract_instructions = []
        for instr in dis.get_instructions(code_obj):
            abstract_op = mapping.get(instr.opname)
            if abstract_op:
                abstract_instructions.append(AbstractInstruction(
                    opcode=abstract_op,
                    arg=instr.arg,
                    offset=instr.offset
                ))
        return abstract_instructions
```

**优点**:
- 反编译器核心逻辑不需要知道具体Python版本
- 添加新版本支持只需更新映射表
- 更清晰的架构分离

**缺点**:
- 初始实现工作量大
- 可能丢失某些版本特定的优化机会
- 需要维护抽象层和具体实现的同步

### 方案B: 插件化版本适配器

为每个Python版本创建独立的适配器：

```python
# bytecode/adapters/

class BaseVersionAdapter(ABC):
    @abstractmethod
    def get_jump_ops(self) -> Set[str]: ...

    @abstractmethod
    def analyze_exceptions(self, cfg) -> ExceptionInfo: ...

    @abstractmethod
    def get_special_region_detectors(self) -> List[Callable]: ...

class Python38Adapter(BaseVersionAdapter): ...
class Python310Adapter(BaseVersionAdapter): ...
class Python311Adapter(BaseVersionAdapter): ...


class AdapterFactory:
    @staticmethod
    def create(version) -> BaseVersionAdapter:
        adapters = {
            (3, 8): Python38Adapter,
            (3, 9): Python39Adapter,
            (3, 10): Python310Adapter,
            (3, 11): Python311Adapter,
        }
        adapter_class = adapters.get(version, Python311Adapter)
        return adapter_class()
```

**优点**:
- 版本间隔离清晰
- 可以针对特定版本做深度优化
- 易于并行开发不同版本的支持

**缺点**:
- 可能存在大量重复代码
- 维护多个适配器的成本高
- 版本间的公共逻辑难以复用

### 方案C: 渐进式统一（当前策略的延续）

继续完善当前的"条件编译 + 特性检测"方案，但增加更多自动化：

```python
# bytecode/unified_analyzer.py (增强版)

class UnifiedAnalyzer:
    def __init__(self, cfg):
        self.cfg = cfg
        self.version = self._detect_version()
        self.adapter = self._create_adapter()

    def _detect_version(self):
        """自动检测字节码版本"""
        if hasattr(self.cfg.code_obj, 'co_exceptiontable'):
            return (3, 11)
        elif any('BACKWARD' in i.opname for b in self.cfg.blocks for i in b.instructions):
            return (3, 10)
        else:
            return (3, 8)

    def analyze(self):
        """统一的分析入口"""
        regions = self.adapter.detect_regions(self.cfg)
        exceptions = self.adapter.analyze_exceptions(self.cfg)
        return AnalysisResult(regions=regions, exceptions=exceptions)
```

## 决策建议

### 短期（Phase 4完成前）
继续使用当前策略，重点：
1. 完善3.11异常表处理的测试覆盖
2. 为所有版本特定代码添加明确的版本注释
3. 扩展skip标记的使用范围，明确记录已知限制

### 中期（Phase 5）
评估并选择以下方向之一：
- 如果团队资源充足 → **方案A（抽象层）**
- 如果需要快速支持新版本 → **方案C（渐进式统一）**
- 如果各版本差异极大 → **方案B（插件化）**

### 长期目标
实现真正的版本无关反编译：
- 输入：任意版本的.pyc文件
- 处理：内部统一为抽象表示
- 输出：正确还原的源代码

## 后果

### 正面影响（如果采用推荐方案）
- ✅ **长期可维护性**: 清晰的版本隔离减少技术债务
- ✅ **新版本支持速度**: 添加Python 3.12+支持从数周缩短至数天
- ✅ **代码质量**: 消除大量的条件编译分支

### 负面影响（当前策略的局限）
- ❌ **代码复杂度持续增长**: 条件编译分支会随版本增加而膨胀
- ❌ **测试矩阵爆炸**: 需要测试 N个版本 × M个功能 的组合
- ❌ **知识分散**: 版本特定的隐含假设散布在代码各处
