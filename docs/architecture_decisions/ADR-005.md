# ADR-005: 为什么功能验证比语法验证更重要

## 状态
✅ 已采纳

## 背景
在反编译器的测试策略上，存在两种主要的验证方法：

### 方法1: 语法验证（Syntax Verification）
检查反编译输出的代码是否是合法的Python代码：

```python
def test_syntax_validity(source_code):
    """语法验证：只检查输出是否可以解析"""
    try:
        ast.parse(source_code)
        return True
    except SyntaxError:
        return False
```

**问题**: 语法正确 ≠ 语义正确

```python
# 原始代码
def example(x):
    if x > 0:
        return x + 1
    else:
        return x - 1

# 反编译结果（语法正确但语义错误）
def example(x):
    if x > 0:
        return x - 1  # ❌ 错误：应该是+1
    else:
        return x + 1  # ❌ 错误：应该是-1
```

这段反编译结果可以通过语法验证，但执行行为完全不同。

### 方法2: 功能验证（Functional Verification）
通过完整的编译→反编译→再编译→字节码对比循环来验证：

```python
def test_functional_equivalence(original_code):
    """功能验证：字节码一致性"""
    original_bc = compile(original_code, '<original>', 'exec')
    decompiled = decompiler.decompile(original_bc)
    recompiled_bc = compile(decompiled, '<decompiled>', 'exec')
    return compare_bytecode(original_bc, recompiled_bc)
```

这种方法能捕获语义级别的错误。

## 决策
**优先使用功能验证作为核心测试标准，语法验证仅作为辅助手段。**

### DecompilationVerifier 设计理念

```python
class DecompilationVerifier:
    """
    反编译功能验证器

    核心流程：
    1. 原始源码 → 编译 → 字节码A (original)
    2. 字节码A → 反编译 → 源码B (decompiled)
    3. 源码B → 编译 → 字节码C (recompiled)
    4. 对比 字节码A 和 字节码C 的一致性
    """

    def verify(self, source_code: str) -> VerificationResult:
        try:
            original_code = self._compile_original(source_code)
            decompiled_source = self._decompile(original_code)
            recompiled_code = self._recompile(decompiled_source)
            differences = self._compare_bytecode(original_code, recompiled_code)

            return VerificationResult(
                is_passing=len(differences) == 0,
                source_code=source_code,
                decompiled_source=decompiled_source,
                differences=differences,
                bytecode_match_rate=self._calc_match_rate(differences)
            )
        except Exception as e:
            return VerificationResult(is_passing=False, error=str(e))
```

## 理由

### 1. 语法验证的局限性

#### 局限性A: 无法检测操作符错误
```python
# 原始: a * b
# 反编译: a + b  （语法正确，语义错误）
```

#### 局限性B: 无法检测控制流错误
```python
# 原始: if condition: A else: B
# 反编译: if condition: B else: A  （then/else交换）
```

#### 局限性C: 无法检测变量名混淆
```python
# 原始: result = calculate(x, y)
# 反编译: result = calculate(y, x)  （参数顺序错误）
```

#### 局限性D: 给出虚假的安全感
项目早期的测试数据：
- **语法验证通过率**: 95%+
- **实际正确率**: 仅40%（大量语义错误未被检测）

这种差距导致团队对代码质量产生误判。

### 2. 功能验证的优势

#### 优势A: 检测语义等价性
功能验证确保反编译后的代码与原始代码在**运行时行为**上一致。

```python
# 测试用例示例
@pytest.mark.parametrize("source", [
    "def f(x): return x + 1",
    "def g():\n for i in range(10):\n  if i % 2 == 0:\n   yield i",
    """
    def h():
        try:
            risky_operation()
        except ValueError as e:
            handle_error(e)
        finally:
            cleanup()
    """,
])
def test_decompilation_produces_equivalent_bytecode(source):
    verifier = DecompilationVerifier()
    result = verifier.verify(source)
    assert result.is_passing, f"Differences found: {result.differences}"
    assert result.bytecode_match_rate > 0.95
```

#### 优势B: 版本无关的验证标准
无论Python版本如何变化，字节码一致性始终是正确的验证目标。

#### 优势C: 自动化回归防护
每次代码修改后自动运行完整的功能验证套件：

```bash
# CI/CD中的测试命令
pytest tests/test_functional_verification.py --verbose --tb=short
```

#### 优势D: 精确的错误定位
当验证失败时，`DecompilationVerifier` 提供详细的差异报告：

```json
{
  "is_passing": false,
  "differences": [
    {
      "offset": 42,
      "expected": {"opname": "BINARY_ADD", "arg": null},
      "actual": {"opname": "BINARY_SUBTRACT", "arg": null},
      "context": "in function 'calculate' at line 5"
    }
  ],
  "bytecode_match_rate": 0.87
}
```

### 3. 测试覆盖率提升数据

| 阶段 | 验证方法 | 覆盖率 | 发现的Bug类型 |
|------|----------|--------|---------------|
| Phase 1 (早期) | 仅语法验证 | 40% | 大量未发现 |
| Phase 2 (过渡) | 语法+抽样功能验证 | 52% | 控制流错误 |
| **Phase 3 (当前)** | **全面功能验证** | **71%** | **细微语义错误** |
| 目标 (Phase 4) | 功能+边界测试 | 85%+ | 极端情况 |

**关键改进点**:

1. **if-elif-else链处理**: 从30%正确率提升至90%+
2. **嵌套循环**: 从25%正确率提升至85%+
3. **try-except-finally**: 从20%正确率提升至75%+
4. **推导式**: 从35%正确率提升至80%+

### 4. 实施策略

#### 分层验证体系

```python
class TestHierarchy:
    """
    L0: 快速冒烟测试 (语法验证)
       - 运行时间: <1分钟
       - 用途: PR预检

    L1: 核心功能验证 (字节码对比)
       - 运行时间: 5-10分钟
       - 用途: 每次提交

    L2: 完整矩阵测试 (所有控制流组合)
       - 运行时间: 30-60分钟
       - 用途: 每日构建

    L3: 边界压力测试 (极端输入)
       - 运行时间: 2-4小时
       - 用途: 发布前验证
    """
```

#### 跳过标记机制
对于已知但不紧急的问题，使用skip标记避免阻塞CI：

```python
@pytest.mark.xfail(reason="Issue #123: ternary in comprehension")
def test_ternary_in_comprehension():
    verifier = DecompilationVerifier()
    source = "[x if x > 0 else -x for x in range(10)]"
    result = verifier.verify(source)
    assert result.is_passing  # 预期失败，但不阻塞CI
```

## 后果

### 正面影响
- ✅ **信心提升**: 通过功能验证的代码有更高的质量保证
- ✅ **回归防护**: 新引入的错误会被立即发现
- ✅ **开发效率**: 精确的错误定位减少调试时间
- ✅ **用户信任**: 可以向用户展示具体的验证通过率数据

### 负面影响
- ❌ **测试执行时间长**: 完整的功能验证需要编译和反编译每个测试用例
- ❌ **测试编写成本高**: 需要为每种控制流模式编写多个变体
- ❌ **对编译器依赖**: 功能验证依赖CPython编译器的一致性
- ❌ **某些优化无法验证**: 编译器优化可能导致相同的源码产生不同字节码
