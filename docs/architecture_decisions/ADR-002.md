# ADR-002: 为什么禁止操作码硬编码过滤

## 状态
✅ 已采纳

## 背景
在AST生成阶段，早期实现中存在大量基于操作码名称（opname）的硬编码过滤逻辑。例如：

```python
# ❌ 禁止的做法（反面案例）
def generate_statements(self, block):
    for instr in block.instructions:
        if instr.opname == 'POP_TOP':
            continue  # 跳过POP_TOP指令
        if instr.opname in ('LOAD_CONST', 'RETURN_VALUE'):
            # 特殊处理这些操作码
            pass
        if 'JUMP' in instr.opname:
            # 跳转指令特殊处理
            pass
```

这种做法导致了严重的维护问题：
1. **版本脆弱性**: Python 3.11引入了新的跳转指令（如 `JUMP_BACKWARD`），所有包含 `'JUMP' in instr.opname` 的代码都需要修改
2. **语义混淆**: 相同的操作码在不同上下文中有不同含义，硬编码无法区分
3. **逻辑分散**: 过滤逻辑散布在生成器的各个方法中，难以统一管理和测试

## 决策
**禁止在AST生成器中使用操作码硬编码过滤。改用区域分析器提供的结构特征检测。**

### 正确做法示例

```python
# ✅ 推荐做法：使用结构特征
def generate_block_statements(self, block):
    semantics = self.region_analyzer.block_semantics.get(block.start_offset)

    # 使用语义标注而非操作码判断
    if semantics and semantics.is_cleanup_block:
        return []  # 清理块不生成代码

    if semantics and semantics.is_continue_target:
        return [{'type': 'Continue'}]

    # 使用有效指令列表（已过滤噪声）
    effective_instrs = semantics.effective_instructions if semantics else block.instructions
    return self._generate_from_instructions(effective_instrs)
```

## 理由

### 1. 操作码是实现细节，不是语义特征
Python字节码的操作码集在不同版本间频繁变化：

| 版本 | 新增/修改的操作码 |
|------|------------------|
| 3.8 | `LOAD_METHOD`, `CALL_METHOD` 替代旧方案 |
| 3.10 | 前向/后向跳转分离 (`JUMP_FORWARD` vs `JUMP_BACKWARD`) |
| 3.11 | 异常表重构，新增 `PUSH_EXC_INFO`, `POP_EXCEPT` 等 |
| 3.12 | 进一步优化，新增 `CHECK_*` 系列指令 |

硬编码操作码名称意味着每次Python版本升级都需要修改反编译器代码。

### 2. 结构特征具有跨版本稳定性
相比之下，控制流结构特征在不同版本间保持稳定：

```python
# ✅ 稳定的特征检测
def is_loop_header(self, block):
    """通过回边检测识别循环头"""
    for successor in block.successors:
        if successor.start_offset < block.start_offset:
            return True
    return False

def is_condition_branch(self, block):
    """通过后继数量和支配边界识别条件分支"""
    return len(block.successors) == 2 and not self.is_loop_back_edge(block)
```

这些特征基于控制流图的结构属性，不依赖于具体的操作码名称。

### 3. 集中管理降低维护成本
将所有操作码相关的处理集中在 `region_analyzer.py` 中：

```python
class RegionAnalyzer:
    NOISE_OPS = frozenset({
        'RESUME', 'NOP', 'CACHE', 'PUSH_NULL',
    })

    CLEANUP_OPS = NOISE_OPS | frozenset({
        'POP_TOP', 'COPY', 'PUSH_EXC_INFO', 'POP_EXCEPT',
        'LOAD_CONST', 'RETURN_VALUE', 'RETURN_CONST', 'RERAISE', 'SWAP',
    })

    def _filter_effective_instructions(self, instructions):
        """集中过滤噪声指令"""
        return [i for i in instructions if i.opname not in self.CLEANUP_OPS]
```

当需要适配新版本时，只需修改这一处定义。

### 4. 语义标注提供更丰富的信息
区域分析器不仅过滤操作码，还提供完整的语义标注：

```python
@dataclass
class BlockSemantics:
    role: BlockRole                    # 角色信息 (LOOP_HEADER, IF_CONDITION等)
    is_cleanup_block: bool             # 是否为清理块
    is_continue_target: bool           # 是否为continue目标
    is_back_edge: bool                 # 是否是循环回边
    effective_instructions: List[Instruction]  # 有效指令列表
    statement_boundaries: List[Tuple[int, int]]  # 语句边界
```

这使得AST生成器可以做出更智能的决策，而不仅仅是"跳过某类操作码"。

## 迁移指南

### 步骤1: 识别现有硬编码
使用以下命令查找违规代码：

```bash
grep -rn "instr.opname ==" core/cfg/region_ast_generator.py
grep -rn "opname in (" core/cfg/region_ast_generator.py
grep -rn "'JUMP'" core/cfg/region_ast_generator.py
```

### 步骤2: 提取到区域分析器
将操作码判断逻辑迁移到 `region_analyzer.py` 的相应方法中。

### 步骤3: 使用语义标注替换
在AST生成器中使用 `block_semantics` 替代直接检查操作码。

### 步骤4: 添加合规性检查
在CI流程中加入自动化审计：

```python
def audit_opcode_hardcoding(filepath):
    with open(filepath) as f:
        content = f.read()

    violations = []
    for line_num, line in enumerate(content.split('\n'), 1):
        if '.opname ==' in line or "'JUMP'" in line:
            violations.append((line_num, line.strip()))

    return violations
```

## 后果

### 正面影响
- ✅ **版本兼容性提升**: 新增Python版本支持时修改点减少80%+
- ✅ **代码可读性改善**: AST生成器的意图更清晰，不再充斥操作码细节
- ✅ **测试更容易**: 可以mock语义标注来测试各种边界情况

### 负面影响
- ❌ **初期重构工作量大**: 需要梳理并迁移现有的硬编码逻辑
- ❌ **对区域分析器依赖增强**: 区域分析器的bug会影响整个系统
- ❌ **调试间接性增加**: 问题可能出在语义标注而非生成逻辑本身
