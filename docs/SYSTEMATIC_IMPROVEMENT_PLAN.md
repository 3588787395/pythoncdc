二、失败根因分析（算法层面）
---------------------------

【核心问题】区域边界计算在嵌套场景下不够精确

以N14为例的字节码结构：
```
@38:  FOR_ITER          ← 外层循环头
@40:  POP_JUMP_IF_FALSE @176  ← if any(...) 条件
@102: GET_ITER           ← 内层循环初始化
@132: FOR_ITER           ← 内层循环头
@134: POP_JUMP_IF_FALSE @166  ← 内层if条件
@154: ...               ← 内层if体
@166: JUMP_BACKWARD @132 ← 内层continue
@168: POP_JUMP_IF_FALSE @176  ← if pos条件
@172: BREAK_LOOP         ← break
@176: JUMP_BACKWARD @38   ← 外层continue
@178: RETURN_VALUE        ← return
```

【问题本质】
当前_analyze_if_region()在收集then_blocks时：
- then_succ = @102 (内层循环初始化)
- _collect_branch_blocks(@102) 收集到 [102, 132, 134, 154, 166]
- 但这缺少了 @168 (if pos) 和 @172 (break)

【为什么缺少】
因为@166 (内层continue) 跳回@132，导致BFS遍历停止，
没有继续收集循环后的@168和@172。

【正确的算法应该是什么？】

根据文档4.2节的"条件区域识别算法"：

function IDENTIFY_IF_REGION(header):
    A, B = header的两个后继
    merge = 最近公共支配后继(A, B)
    then_blocks = 从A可达且不经过B和merge的所有块

关键在于**"不经过B"的定义**：
- 当前实现：基本块级别的路径检查
- 应该是：如果A属于某个LoopRegion，则该LoopRegion的出口也属于then
